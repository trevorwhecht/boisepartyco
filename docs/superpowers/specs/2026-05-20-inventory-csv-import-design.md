# Inventory CSV Import/Export — Design Spec

**Date:** 2026-05-20
**Status:** Approved

---

## Goal

Allow admins to download a pre-filled CSV template for inventory quantities, edit it offline, and re-upload to apply bulk updates. Blank cells are skipped (not zeroed), making it safe for partial updates and initial inventory setup.

---

## Scope

- **Items** (non-tent categories): template scoped to the active category tab
- **Tent Parts** (Tents tab): template covers all tent parts
- **Tent Configurations**: excluded — no qty to manage directly
- **Admin only**: both download and upload are admin-only actions

---

## CSV Formats

### Items (active category)

```
id,name,sku,qty
1,Round Table (60in),TBL-RND-60,25
2,Folding Chair,CHR-FOLD-01,100
```

### Tent Parts (Tents tab)

```
id,name,part_type,qty
1,8×8 Panel,panel,40
2,Center Crown,crown,12
```

**Rules:**
- `id` is the primary match key — never modified by user
- `name`, `sku`, `part_type` are read-only context — ignored on upload
- `qty` blank → skip this row entirely (no update)
- `qty` present → must be a non-negative integer; invalid values are reported as row errors
- Valid rows still apply even if other rows have errors

---

## Architecture

### Approach: Client-side CSV parse + bulk import endpoint

Parse CSV in the browser with a plain string split (no npm dep — the template format is controlled and won't have quoted commas). Send a JSON array of `{ id, qty }` to a new bulk import route. One round-trip per upload.

### New API Routes

**`POST /api/admin/inventory/items/import`**
- Auth: admin only (403 for employee)
- Body: `{ rows: { id: number; qty: number }[] }` — already filtered (blanks removed client-side)
- Logic: for each row, `prisma.item.update({ where: { id }, data: { qty } })`
- Response: `{ data: { updated: number; skipped: number; errors: string[] }; error: null }`

**`POST /api/admin/inventory/tent-parts/import`**
- Auth: admin only (403 for employee)
- Body: `{ rows: { id: number; qty: number }[] }`
- Logic: for each row, `prisma.tentPart.update({ where: { id }, data: { qty } })`
- Response: same shape as items import

Both routes process all rows independently — a failure on one row does not abort the others.

### Modified UI

**`Dashboard-InventoryView.tsx`**

Add two small buttons to the top-right of the inventory header (visible only when `role === "admin"`):

- **`↓ Template`** — generates CSV from in-memory data and triggers browser download
- **`↑ Upload CSV`** — hidden `<input type="file" accept=".csv">` triggered by the button

Both buttons adapt to the active tab:
- `activeTab === "tent"` → tent parts CSV
- Any other tab → items CSV for that category

The component already holds `categories` (with `id` and `slug`) and `activeTab`. The tab content children (`CategoryTab`, `TentsTab`) have their own local data state. To avoid prop-drilling or refetching, the CSV template is generated by fetching fresh data at download time (one GET call when the user clicks `↓ Template`).

### Client-side CSV Logic (in `Dashboard-InventoryView.tsx`)

**Download:**
1. Fetch current data (`/api/admin/inventory/items?categoryId=X` or `/api/admin/inventory/tent-parts`)
2. Build CSV string from response
3. `URL.createObjectURL(new Blob([csv], { type: "text/csv" }))` + anchor click
4. Filename: `inventory-{category-slug}-{date}.csv` or `inventory-tent-parts-{date}.csv`

**Upload:**
1. Read file via `FileReader`
2. Parse: split by newline, skip header row, split each row by comma
3. Validate header row matches expected template (`id,name,sku,qty` for items; `id,name,part_type,qty` for tent parts) — mismatch → error toast, abort
4. For each data row: parse `id` (col 0) and `qty` (col 3)
5. Skip rows where `qty` cell is blank or whitespace-only
6. Validate `qty` is a non-negative integer — collect row errors
7. If no valid rows after filtering: show error toast, abort
8. POST `{ rows: validRows }` to the appropriate import route
9. Toast: `"N items updated, M skipped"` on success; `"N updated. Errors: row 4 invalid qty, …"` if partial errors

## Error Handling

| Scenario | Behavior |
|---|---|
| Blank qty cell | Skip row, count as skipped |
| Non-integer qty | Row error — reported in toast, other rows still applied |
| Negative qty | Row error — reported in toast |
| id not found in DB | Prisma throws — caught per-row, reported as error |
| Wrong file format | Client detects missing/wrong headers → error toast before POST |
| Network failure | Error toast |
| Employee tries to upload | 403 → error toast |

## Out of Scope

- Tent configuration quantities (derived from parts)
- Exporting/importing isActive or primaryImageUrl via CSV
- Multi-category bulk export (one tab = one template)
- CSV validation preview before applying
- Employee download access
